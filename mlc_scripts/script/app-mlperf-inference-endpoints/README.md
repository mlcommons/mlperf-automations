# app-mlperf-inference-endpoints

Benchmark an LLM inference **endpoint** (an OpenAI/SGLang/VideoGen-compatible
HTTP server) with the MLCommons
[inference-endpoint](https://github.com/mlcommons/endpoints) load generator,
driven through MLC automation.

This script assembles and runs `inference-endpoint benchmark <offline|online>`,
then parses the result into MLC environment variables so it can be stitched into
larger workflows.

## What it depends on

| Dependency | Purpose |
|------------|---------|
| `detect,os`, `detect,cpu` | platform detection (auto-disables CPU pinning off Linux) |
| `get,python3` | host Python |
| `get,mlperf,endpoints` | installs the `inference-endpoint` package into a dedicated venv |

`get,mlperf,endpoints` in turn pulls `get,mlperf,endpoints,src` (the source
checkout). Each script is independently runnable and composable.

## Quick start (self-contained, no real server)

The `_echo-server` variation launches the bundled echo server, points the
benchmark at it, and defaults to the bundled dummy dataset:

```bash
# Offline (max throughput)
mlcr app,mlperf,inference,endpoints,_offline,_echo-server --num_samples=200

# Online (sustained QPS, Poisson arrivals)
mlcr app,mlperf,inference,endpoints,_online,_poisson,_echo-server \
  --target_qps=50 --duration=10s

# Online (fixed concurrency)
mlcr app,mlperf,inference,endpoints,_online,_fixed-concurrency,_echo-server \
  --concurrency=16 --duration=10s
```

## Config-file driven runs (from-config)

Mirroring the inference `user.conf` flow, you can drive the benchmark from a YAML
config (generated by
[generate-mlperf-endpoints-conf](../generate-mlperf-endpoints-conf/README.md)):

```bash
# 1) Generate, inspect, then run an existing config
mlcr generate,mlperf,endpoints-conf --conf_type=offline --model=my-model \
  --dataset=/data/prompts.jsonl --prompt_column=text --conf_path=run.yaml
mlcr app,mlperf,inference,endpoints --config=run.yaml --endpoints=http://host:8000

# 2) Or generate + run in one step with the _from-config variation
mlcr app,mlperf,inference,endpoints,_from-config \
  --model=my-model --dataset=/data/prompts.jsonl --prompt_column=text \
  --endpoints=http://host:8000 --conf_type=online --target_qps=100
```

In `_from-config` mode the inputs flow to the generator (which writes the YAML)
and the app runs it via `benchmark from-config`. The script aligns the report
directory so results land in `--report_dir`.

## Benchmarking a real endpoint

```bash
mlcr app,mlperf,inference,endpoints,_offline \
  --endpoints=http://your-host:8000 \
  --model=meta-llama/Llama-3.1-8B-Instruct \
  --dataset=/path/to/data.jsonl \
  --prompt_column=text_input \
  --num_samples=1000
```

## Variations

| Group | Variations | Notes |
|-------|------------|-------|
| `mode` | `offline` (default), `online` | offline = max throughput; online = sustained load |
| `api-type` | `openai` (default), `sglang`, `videogen` | request/response protocol |
| `load-pattern` | `poisson`, `fixed-concurrency`, `multi-turn` | online only; inferred from inputs if omitted |
| `cpu-affinity` | `cpu-affinity`, `no-cpu-affinity` | NUMA pinning (Linux only; auto-off elsewhere) |
| (flag) | `echo-server` | launch the bundled echo server for local testing |
| (flag) | `from-config` | generate a YAML config from the inputs and run it via `benchmark from-config` |

## Key inputs

| Input | Description |
|-------|-------------|
| `--endpoints` / `--api_endpoint` | endpoint base URL |
| `--model` | model name the endpoint serves (required unless `_echo-server`) |
| `--dataset` | dataset path (required unless `_echo-server`) |
| `--prompt_column` | dataset column holding the prompt (maps to `parser.prompt`) |
| `--num_samples` | number of dataset samples to use |
| `--target_qps` | target QPS (online/poisson) |
| `--concurrency` | concurrent requests (online/fixed-concurrency) |
| `--num_workers` / `--workers` | client worker processes (`-1`=auto) |
| `--duration` | minimum duration (`600s`, `10m`, or ms) |
| `--config` | run a pre-existing YAML config via `from-config` |
| `--max_output_tokens`, `--streaming`, `--tokenizer`, `--api_key`, `--timeout` | passed through |
| `--report_dir` / `--output_dir` | where results are written |

## Outputs

Result files are written to `--report_dir` (`results.json`,
`result_summary.json`, `report.txt`, `metrics/final_snapshot.json`, …). The
script also exports:

| Env key | Meaning |
|---------|---------|
| `MLC_MLPERF_ENDPOINT_QPS` | achieved QPS |
| `MLC_MLPERF_ENDPOINT_SAMPLES_TOTAL` / `_SUCCESSFUL` / `_FAILED` | sample counts |
| `MLC_MLPERF_ENDPOINT_ELAPSED_TIME` | wall-clock seconds |
| `MLC_MLPERF_ENDPOINT_LATENCY_MEAN_NS` / `_P50_NS` / `_P99_NS` | latency stats (perf runs) |
| `MLC_MLPERF_ENDPOINT_RESULTS_FILE` / `_REPORT_DIR` | result locations |

## Tests

End-to-end workflow tests (offline, online poisson, online concurrency, and a
negative case) live in [tests/test_endpoints_workflow.py](tests/test_endpoints_workflow.py)
and run against the echo server:

```bash
pytest -q script/app-mlperf-inference-endpoints/tests/test_endpoints_workflow.py
```

The same scenarios are runnable via the MLC-native runner:

```bash
mlc test script --tags=app,mlperf,inference,endpoints
```
